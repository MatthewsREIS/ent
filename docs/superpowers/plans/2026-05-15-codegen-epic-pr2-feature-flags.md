# Codegen Epic PR 2 — Feature Flags Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Give consumers opt-out knobs to skip generating update and delete builders (globally via feature flags, per-entity via a schema annotation), so read-mostly or read-only data models don't pay for code they'll never use.

**Architecture:** Two existing-pattern extensions:
1. Two new `Feature` entries (`FeatureNoUpdate`, `FeatureNoDelete`) in `entc/gen/feature.go`, following the pattern of `FeaturePrivacy` / `FeatureIntercept` / `FeatureEntQL`. Templates that emit update/delete artifacts gate themselves on `{{ if not ($.FeatureEnabled "no-update") }}` etc.
2. One new schema annotation (`entc.ReadOnly`) exposed via the existing `schema.Annotation` interface. Templates check the per-entity annotations alongside the global feature flags.

**Tech Stack:** Go stdlib. No new dependencies.

**Spec reference:** `docs/superpowers/specs/2026-05-15-ent-codegen-reduction-epic.md` §4 PR 2 (the section currently labeled "PR 2 — codegen-epic/02-feature-flags" after the 2026-05-15 renumbering).

**Branch:** stay on `worktree-wiggly-singing-pancake` (epic policy — no separate git-spice branch, no push, no PR).

## Scope cuts vs. the spec

The spec's PR 2 lists four flags: `FeatureNoMutations`, `FeatureNoUpdate`, `FeatureNoDelete`, `FeatureNoGraphQL`. This plan ships **two of them** (`NoUpdate`, `NoDelete`) plus the `ReadOnly` per-entity annotation. The other two are explicitly deferred with rationale:

- **`FeatureNoMutations` deferred.** Mutation types are deeply integrated: create/update/delete builders all hold a `*<Entity>Mutation`. Cleanly removing mutations requires removing those builders too — which `NoUpdate`/`NoDelete` already accomplishes. A standalone `NoMutations` flag adds complexity without much marginal value over the combination of `NoUpdate` + `NoDelete` (which leaves create-only mutations, plenty for a write-once data model). If a consumer needs strict no-writes, they can use the `ReadOnly` annotation per entity, which today suppresses update+delete; future work can extend it to suppress create too.
- **`FeatureNoGraphQL` deferred.** GraphQL bindings are generated by `entgo.io/contrib/entgql`, a separate codegen extension that runs on top of `entc.Generate`. A flag in `ent` itself can't suppress entgql output without cross-repo coordination. This belongs in the entgql repo as `entgql.NoGenerate(...)` or similar, not in ent's feature system.

If the team disagrees, the deferred flags can be added as a small follow-up PR after this one lands.

```go
// cleanupSuffix walks dir non-recursively and removes every regular file whose
// name ends with suffix. Used by FeatureNoUpdate / FeatureNoDelete to clear
// stale builders left over from a previous regen.
func cleanupSuffix(dir, suffix string) error {
	entries, err := os.ReadDir(dir)
	if err != nil {
		if os.IsNotExist(err) {
			return nil
		}
		return err
	}
	for _, e := range entries {
		if e.IsDir() {
			continue
		}
		if !strings.HasSuffix(e.Name(), suffix) {
			continue
		}
		if rmErr := os.Remove(filepath.Join(dir, e.Name())); rmErr != nil {
			return rmErr
		}
	}
	return nil
}
```

Add `"strings"` to the imports of `feature.go` if it isn't already there.

package entc

import "entgo.io/ent/schema"

// readOnlyAnnotation is the carrier type for the ReadOnly schema annotation.
// It is exported via the ReadOnly() constructor; users do not construct it
// directly. The Name() method returns "ReadOnly" so templates can key on it
// via $.Annotations.ReadOnly.
type readOnlyAnnotation struct{}

// Name implements schema.Annotation.
func (readOnlyAnnotation) Name() string { return "ReadOnly" }

// ReadOnly returns a schema annotation that suppresses generation of
// update and delete builders for the annotated entity, regardless of the
// global FeatureNoUpdate / FeatureNoDelete flags. Use it like:
//
//	func (User) Annotations() []schema.Annotation {
//	    return []schema.Annotation{entc.ReadOnly()}
//	}
//
// The annotation has no runtime semantics — it only affects codegen output.
func ReadOnly() schema.Annotation {
	return readOnlyAnnotation{}
}
```

	if ok, _ := g.Config.FeatureEnabled(FeatureNoUpdate.Name); ok {
		out := make([]string, len(g.Nodes))
		for i, n := range g.Nodes {
			out[i] = n.Name
		}
		return out
	}
	var out []string
	for _, n := range g.Nodes {
		if _, has := n.Annotations["ReadOnly"]; has {
			out = append(out, n.Name)
		}
	}
	return out
}

// SkipDeleteTypes mirrors SkipUpdateTypes for the Delete builder.
func (g *Graph) SkipDeleteTypes() []string {
	if ok, _ := g.Config.FeatureEnabled(FeatureNoDelete.Name); ok {
		out := make([]string, len(g.Nodes))
		for i, n := range g.Nodes {
			out[i] = n.Name
		}
		return out
	}
	var out []string
	for _, n := range g.Nodes {
		if _, has := n.Annotations["ReadOnly"]; has {
			out = append(out, n.Name)
		}
	}
	return out
}

---
id: feature-flags-no-update-delete
title: Opt-out feature flags (no-update, no-delete)
---

## When you'd use these

For consumers with read-mostly or read-only data models (audit logs, event
sourcing read models, analytics tables, snapshot caches), the generated
`*_update.go` and `*_delete.go` files are dead weight: they're never called
by application code, but they're compiled by every `go build`.

PR 2 of the codegen-reduction epic adds two global feature flags and one
per-entity annotation that let consumers shrink generated output by opting
out of update and delete builders.

## Global flags

Enable on the CLI:

```
ent generate --feature no-update,no-delete ./ent/schema
```

Or in your generator main:

```go
err := entc.Generate("./schema", &gen.Config{
    Features: []gen.Feature{gen.FeatureNoUpdate, gen.FeatureNoDelete},
})
```

When enabled, NO `*_update.go` / `*_delete.go` files are generated for
any entity. The `<Entity>Client` types also lose their `Update`,
`UpdateOne`, `UpdateOneID`, `Delete`, `DeleteOne`, and `DeleteOneID`
methods.

## Per-entity annotation

When only some entities are read-only, use the annotation:

```go
import (
    "entgo.io/ent"
    "entgo.io/ent/entc"
    "entgo.io/ent/schema"
)

type AuditLog struct {
    ent.Schema
}

func (AuditLog) Annotations() []schema.Annotation {
    return []schema.Annotation{entc.ReadOnly()}
}
```

`AuditLog` gets a Query builder and a Create builder, but no Update or
Delete. Other entities are unaffected.

## Composition

The annotation is additive with the global flags. If `FeatureNoUpdate`
is on globally, every entity is skipped regardless of annotations.
If only some entities have `ReadOnly`, only those are skipped.

## What's NOT covered

- **Mutations.** This PR keeps the per-entity Mutation type because
  Create still needs it. A future PR may add `FeatureNoMutations` if
  there's demand for a truly write-nothing data model.
- **GraphQL bindings.** GraphQL is generated by `entgo.io/contrib/entgql`,
  a separate extension. A `FeatureNoGraphQL` flag belongs in that repo,
  not in ent itself.
- **Create builders.** There is no `FeatureNoCreate` because entities
  with no Create are unusable (you can't put data in). If you have such
  an entity, it should not be an ent schema.
```
